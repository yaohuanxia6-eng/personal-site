# 个站介绍 — 面试逐字稿

## 开场：为什么做个站

找工作投简历，大家都是一份 PDF，面试官每天看几十份。我就想，有没有一种方式能在面试前就让人觉得"这个人是真动手做过东西的"。所以我搭了这个个人站点，把经历、项目、作品全放上去，投简历附一个链接。

## 整体设计思路

风格上我选了偏复古文艺的方向——米色纸张质感背景，衬线字体搭配等宽字体，像旧杂志加终端的混搭。首屏有台纯 CSS 搭建的 3D 像素风复古电脑，屏幕上跑打字机效果，右下角 AI 机器人在浮动眨眼。这个设计是想说：我既懂传统产品方法论，也拥抱 AI 新技术。

## 核心产品：粘豆包

个站上最有深度的项目是**粘豆包**——一款 AI 情绪陪伴产品，从 0 到 1 全栈独立开发并上线。

**我的想法**：市面上的 AI 聊天产品（Replika、Character.AI）大多做"无目的闲聊"，聊完就忘，没有成长价值。我想做的是：让每次对话都有价值沉淀——情绪记录、记忆积累、行动建议。

**技术选型与架构**：
- 前端 Next.js 14 + Tailwind，后端 FastAPI + MySQL，认证 Supabase
- AI 用 Kimi API（moonshot-v1-8k），图片识别用 Kimi Vision
- 部署在腾讯云，Nginx 反向代理，全程 SCP 手动部署（服务器无法访问 GitHub）

**开发中的关键卡点**：

1. **Supabase JWT 算法坑**：用户 JWT 是 ES256 不是 HS256，后端用 JWT Secret 验证一直失败。排查后发现要从 JWKS 端点获取公钥验证。这个坑官方文档没明确写。

2. **图片识别超时**：Next.js API Route 每次调 `supabase.auth.getUser()` 要发请求到海外 Supabase 服务器（1-3 秒），叠加 Kimi Vision 处理时间，手机端经常超时。我的解决方案：跳过网络验证，直接在本地解码 JWT payload 获取 user_id，同时把图片从原尺寸压缩到 600px、只发最后一条消息给 vision API。三管齐下，响应时间从 8-10 秒降到 3-4 秒。

3. **记忆系统设计**：一开始每 6 条消息就调 AI 提取"关键事实"，结果记忆变得很碎片——"用户对 AI 感兴趣""用户发了一张图"，没有价值。改为每日只总结一次，AI 生成日记式的 100-200 字总结，按日期存储，像用户的情绪日记本。

4. **AI 输出格式**：Kimi 经常输出 markdown（`**加粗**`、编号列表），在手机端看就是一大坨。我做了双重保险——后端 prompt 明确禁止 markdown 并给出正确格式示例，前端 ChatBubble 再做一层智能清理和分段渲染。

**产品亮点**：
- MBTI 个性化：16 型人格各自的聊天风格和 system prompt
- 认知重构工具：用户填写后调用 AI 做真实的认知偏差分析，不是硬编码模板
- 微行动系统：AI 建议具体的 5-15 分钟小行动，跟踪完成率
- 回归问候：离开 App 超过 3 分钟回来，AI 会用不同的话主动打招呼

## 与 AI 协作的方式

这个站从头到尾都是我和 AI 一起做的，但所有产品决策是我做的——站点结构、功能优先级、视觉风格、技术选型。AI 不会替我想"面试官看什么"或"记忆系统该怎么设计"，这是产品经理的判断。

我跟 AI 的协作方式像带一个高效的初级开发：我给具体需求和方向，它执行和输出代码；遇到问题它的第一个方案不一定对，需要我有技术理解力来引导。比如图片识别超时的问题，AI 最初建议"增加超时时间"，但真正的解决方案是去掉不必要的网络请求——这个判断需要我理解整个认证链路。

每次改完一轮，我会让 AI 更新项目档案，记录踩坑、架构变更、部署命令。新对话窗口读完文档就能接上上下文，不从头讲。

## 总结

这个个站体现了三个核心能力：**结构化思维**——从产品定位到 PRD 到信息架构，全流程文档沉淀；**落地能力**——不是停在原型阶段，是真正上线可访问的全栈产品；**AI 协作力**——不是甩手让 AI 干活，而是我负责决策和质量把关，AI 负责执行，像带团队一样高效协作。
