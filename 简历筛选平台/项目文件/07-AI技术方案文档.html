§ 01

AI 在本系统中的价值定位

业务痛点分析

中国电科五十二研究所每年面向社会招聘 120-150 个岗位，涵盖后端开发、前端开发、产品经理、UI 设计、测试工程师等多个技术方向。当前由 16 人 HR 团队负责全流程招聘管理。经过深入调研，我们识别出以下核心痛点：

痛点编号 问题描述 影响程度 量化数据
P1 HR 每日简历筛选耗时过长，人工逐份阅读效率低下 严重 人均 3.5 小时/天，占工作时间 44%
P2 基于关键词的简单匹配导致大量不匹配候选人进入面试 严重 面试转化率仅 5%，80% 面试无效
P3 不同 HR 对同一简历的评判标准不一致 中等 同一简历不同 HR 打分差异达 30%+
P4 优质候选人因处理延迟被竞争对手抢先录用 中等 平均简历处理周期 3-5 天
P5 多渠道简历汇总困难，存在重复处理和遗漏 中等 约 15% 简历被重复筛选或遗漏

AI 解决方案定位

本系统引入大语言模型（LLM）驱动的语义理解能力，从根本上替代传统关键词搜索模式。AI 不再停留在字面匹配，而是深入理解简历内容的语义信息，评估候选人与岗位的真实匹配度。

核心理念 传统筛选 = 关键词命中率统计；AI 筛选 = 多维度语义理解 + 结构化评分 + 证据溯源。AI 不是替代 HR，而是作为 HR 的智能助手，将重复性筛选工作自动化，让 HR 聚焦于高价值的面试和人才评估环节。

三大核心 AI 能力

能力一：智能简历解析

  • 多格式支持：PDF、Word（docx）、图片（JPG/PNG）、扫描件
  • OCR 识别：基于 Tesseract 的中英文混合文档识别
  • 结构化提取：将非结构化文本转为标准化 JSON 数据
  • 字段覆盖：姓名、联系方式、教育背景、工作经历、技能列表、项目经验

能力二：多维度 AI 评分

  • 语义匹配：基于 LLM 理解简历与 JD 的深层语义关联
  • 五维评估：技术深度、项目经验、行业理解、协作能力、学习潜力
  • 证据溯源：每个评分维度均引用简历原文作为依据
  • 红线机制：一票否决规则保障最低准入标准

能力三：智能匹配与排序

  • 全局排序：同一岗位的所有候选人按综合得分自动排序
  • 推荐分层：强烈推荐 / 推荐 / 建议复核 / 不推荐四级分层
  • 相似推荐：基于向量检索发现相似优质候选人
  • 人才库检索：语义搜索历史候选人库，快速召回往期优质人选

预期成效

60% 筛选时间缩减
5%→10% 面试转化率提升
≥80% AI 推荐准确率
3天→1天 处理周期缩短

按照 16 人 HR 团队、人均每日 3.5 小时筛选工作计算：年度节省工时约 16 x 3.5 x 0.6 x 250 = 8,400 小时，折合约 4.2 人年 的人力释放，使 HR 能够投入更多精力在面试评估、候选人沟通和雇主品牌建设等高价值工作上。

§ 02

大模型选型与对比

候选模型对比

在综合考虑国产化合规要求、中文理解能力、部署灵活性和成本等因素后，我们对主流大语言模型进行了全面评估，最终选定 DeepSeek-V3 和通义千问 Qwen-Max 作为主备模型组合。

评估维度 DeepSeek-V3（主模型） 通义千问 Qwen-Max（备用） GPT-4o（参考）
厂商 深度求索（国产） 阿里巴巴（国产） OpenAI（美国）
API 成本 ~0.5 元/万 tokens ~1 元/万 tokens ~7 元/万 tokens
响应延迟 ~800ms（首 token） ~900ms（首 token） ~1200ms（首 token）
中文理解 优秀（中文原生训练） 优秀（阿里原生中文模型） 良好（多语言通用）
推理能力 良好 优秀 优秀
JSON 输出稳定性 良好（偶需重试） 优秀（原生 JSON Mode） 优秀
私有化部署 支持（开源可部署） 支持（阿里云专属） 不支持
数据合规 国产合规 国产合规 数据出境风险
适用场景 日常批量筛选、P5 及以下岗位 复杂评估、P6+ 高级岗位 ——

为什么不选 GPT-4o

国产化合规要求 作为中国电科下属研究所，人力资源数据属于敏感信息范畴。根据《数据安全法》和《个人信息保护法》要求，简历数据不得出境至境外服务器。GPT-4o 虽然综合能力优秀，但其 API 数据必须经由境外服务器处理，存在数据主权风险，不符合国产化合规要求。DeepSeek-V3 和通义千问均支持国内私有化部署，数据全程不出内网。

模型路由策略

根据岗位级别和复杂度，系统自动选择最合适的模型进行评分：

岗位级别 默认模型 路由原因 可覆盖
P5 及以下（初级/中级） DeepSeek-V3 评估维度相对明确，侧重硬技能匹配，DeepSeek 性价比最优 是
P6+（高级/专家/管理） 通义千问 Qwen-Max 高级岗位需评估软技能、管理能力、战略思维等复杂维度 是
手动指定 HR 自选 业务负责人可针对特定岗位手动指定评估模型 ——

故障降级策略

主模型调用 → 超时(10s)? → 重试一次 → 仍失败? → 切换备用模型 → 备用也失败? → 标记"解析失败"

故障降级全程自动执行，无需人工干预。切换备用模型时，评分结果会标注实际使用的模型版本，确保评分可追溯。系统每分钟检测模型可用性，一旦主模型恢复，自动切回主模型。

§ 03

AI 评分维度体系

五维评分模型

AI 评分采用五维度加权模型，每个维度独立评分（0-10 分），最终加权计算综合得分。各维度权重可由业务负责人按岗位需求灵活调整。

维度 默认权重 评估内容 提取方法
核心技术栈深度 30% 候选人核心技术栈与 JD 要求的匹配度，技术深度而非广度。关注是否有相关技术的深入实践经验，而非简单列举关键词。 关键词提取 + LLM 语义匹配 JD 技术要求
高并发/项目经验 25% 实际项目的规模指标：DAU、QPS、团队规模、数据量级；项目中承担的角色和具体贡献。 NER 提取数字指标 + LLM 评估项目复杂度
行业业务理解 15% 候选人所在行业与目标岗位的行业相关性。政企/国防优先，ToB SaaS 次优，纯 ToC 需额外评估迁移能力。 行业关键词 + LLM 判断领域相关性
团队协作与产出 15% 跨团队协作经验、技术方案推动能力、管理信号（带队人数、导师经历）、沟通表达能力线索。 LLM 分析简历中的协作描述和领导力信号
学习潜力与自驱力 15% 教育背景（学历/学校层次）、技术博客/GitHub 活跃度、开源贡献、技术社区参与、简历质量和专业度。 字段提取 + LLM 评估成长轨迹
权重可配置 以上权重为系统默认值。业务负责人在创建或编辑岗位时，可以自定义各维度的权重、新增专属评估维度、调整提取规则。例如：前端岗位可将"核心技术栈深度"权重提高至 40%，同时降低"行业业务理解"至 10%。所有调整均实时生效，历史评分不受影响。

红线过滤规则（一票否决）

在 AI 评分之前，系统先执行红线过滤。触发任一红线规则的候选人将被直接标记为"不推荐"，不再进入 AI 评分流程。

红线类型 规则描述 示例 可配置
最低经验年限 工作年限低于岗位最低要求 JD 要求 3 年以上，候选人仅 1 年 是（按岗位设置）
学历底线 学历不满足岗位最低要求 JD 要求本科及以上，候选人为专科 是（按岗位设置）
工作稳定性 近 3 年内跳槽次数超过阈值 3 年内换了 4 家公司 是（默认阈值 3 次）
黑名单 候选人在系统黑名单中 曾因严重问题被拒录的候选人 由 HR 管理员维护

评分等级划分

≥85 强烈推荐
≥75 推荐
≥60 建议复核
<60 不推荐
等级 分数区间 HR 操作建议 预期占比
强烈推荐 85-100 分 优先安排面试，建议 48 小时内联系候选人 ~10-15%
推荐 75-84 分 纳入面试候选池，可按业务节奏安排 ~20-30%
建议复核 60-74 分 HR 人工复核 AI 评分理由，决定是否面试 ~25-35%
不推荐 0-59 分 默认不进入面试流程，HR 可"强捞"例外处理 ~30-40%
§ 04

数据安全与隐私合规

AI 调用前数据脱敏

简历数据在发送给大模型 API 之前，必须经过严格的脱敏处理。脱敏在本地服务端完成，确保敏感信息不会离开内部系统。

脱敏字段 处理方式 是否必选 示例
姓名 替换为"某候选人" 必选 "张三" → "某候选人"
手机号 完全隐藏 必选 "138****1234" → 移除
邮箱 完全隐藏 必选 "xxx@xxx.com" → 移除
证件照/头像 移除图片 必选 不发送任何图片至模型
现单位名称 替换为"某公司"或行业描述 可选 "阿里巴巴" → "某互联网大厂"
院校名称 替换为院校层次 可选 "清华大学" → "某 985 高校"
可选脱敏说明 现单位名称和院校名称的脱敏为可选项，由系统管理员在后台配置。开启后可进一步降低信息泄露风险，但可能影响"行业业务理解"和"学习潜力"维度的评分精度。建议根据实际安全等级需求决定是否开启。

内网私有化部署

部署架构安全保障

  • 零数据出境：所有服务部署在研究所内网，简历数据全生命周期不出内网
  • 模型 API 内网化：DeepSeek-V3 支持本地部署，通义千问可通过阿里云专属区域部署
  • 网络隔离：AI 服务与外网物理隔离，仅通过内部 API 网关对接前端系统
  • 加密传输：内网服务间通信采用 mTLS 加密，防止中间人攻击

审计与追溯

每一次 AI 操作都会留下完整的审计记录，确保操作可追溯、可审查。

记录字段 说明 示例值
操作时间 精确到毫秒的时间戳 2026-04-04 14:30:25.123
操作人 触发 AI 评分的 HR 工号 HR-2024-008
操作类型 解析/评分/重评/强捞 AI_SCORE
模型版本 实际调用的模型和版本号 deepseek-v3-20260401
输入摘要 发送给模型的脱敏后 prompt 哈希 sha256:a3f8c2...
输出结果 模型返回的完整评分 JSON {'score': 82, 'tier': '推荐', ...}

数据留存策略

简历数据采用 AES-256 加密存储，留存周期可由管理员配置。默认策略为：活跃候选人数据永久保留，已关闭岗位相关数据保留 2 年后自动归档加密，归档数据保留 5 年后彻底销毁。所有销毁操作均生成不可篡改的销毁日志。

§ 05

AI 准确率与人工修正闭环

准确率定义与目标

AI 推荐准确率是衡量系统核心价值的关键指标。我们采用以下公式进行计算：

// 准确率计算公式 准确率 = (AI推荐且HR通过 + AI淘汰且HR淘汰) / 总评估数 × 100% // 其中： // AI推荐且HR通过 = AI 给出"推荐"或"强烈推荐"且 HR 确认通过面试的数量 // AI淘汰且HR淘汰 = AI 给出"不推荐"且 HR 未做例外处理（强捞）的数量 // 总评估数 = 评估周期内 AI 处理的简历总数
指标 定义 初期目标 优化目标
综合准确率 AI 判断与 HR 判断一致的比例 ≥80% ≥86%
假阳性率 AI 推荐但 HR 否决（高分误判） ≤15% ≤10%
假阴性率 AI 不推荐但 HR 强捞（低分遗珠） ≤8% ≤5%

人工修正闭环机制

HR 的每一次操作反馈都是模型优化的宝贵数据。系统通过"人工修正 → 数据采集 → 规则调整 → 验证上线"的闭环持续提升 AI 准确率。

HR 审核 AI 结果 → 修正操作（强捞/否决） → 系统记录修正数据 → 每日凌晨分析 → 调整维度权重 → 验证集回归测试 → 自动上线/回滚

每日自学习机制

每日凌晨 2:00 AM 自动执行

  • 采集：收集前一天所有 HR 修正操作，包括强捞低分候选人、否决高分候选人、手动调分
  • 分析：识别哪些评分维度存在系统性偏差，例如"技术深度"维度是否持续偏高
  • 调整：对偏差维度的权重进行微调（单次调整幅度不超过 ±2%，防止过度拟合）
  • 验证：将调整后的规则在 200 份标注数据集上运行回归测试
  • 决策：仅当准确率提升时自动部署新规则，否则保持原规则不变

A/B 测试灰度发布

对于较大的规则变更（如新增评分维度、修改 Prompt 模板），采用灰度发布策略逐步验证：

10% 流量 → 观察 3 天 → 30% 流量 → 观察 3 天 → 50% 流量 → 观察 3 天 → 100% 全量

灰度期间，系统同时记录新旧规则版本的评分结果，产品团队可在后台查看对比报告，随时决定加速推进或回滚。

API 调用成本估算

基础假设

参数 数值 说明

年度处理简历数 ~500 份 含多渠道去重后的有效简历
单份简历输入 tokens ~2,000 tokens 脱敏后简历文本 + JD 文本 + System Prompt
单份简历输出 tokens ~800 tokens 五维度评分 + 证据 + 风险提示的 JSON
主模型使用比例 ~80%（DeepSeek-V3） P5 及以下岗位走主模型
备模型使用比例 ~20%（通义千问） P6+ 岗位走备用模型

成本明细

模型 简历数量 tokens/份 单价 年度费用
DeepSeek-V3 400 份 2,800 0.5 元/万 tokens 56 元
通义千问 Qwen-Max 100 份 2,800 1.0 元/万 tokens 28 元
重评 / 重试 ~50 份（估计） 2,800 混合单价 ~10 元
年度 API 总成本 ~94 元
成本对比分析 年度 API 成本不足 100 元，而 HR 团队每年节省约 8,400 小时工时。按人力成本估算，仅人工筛选时间的节省价值就超过 30 万元/年。投入产出比极为可观，AI 能力的引入在成本维度几乎可以忽略不计。

成本预警机制

虽然当前预估成本极低，系统仍设置了多级成本预警机制以应对异常情况：

  • 日预警：单日 API 调用费用超过 5 元触发邮件通知
  • 周预警：单周累计超过 20 元触发管理员通知
  • 月熔断：单月累计超过 100 元自动暂停非紧急 AI 调用，仅保留手动触发

§ 07

整体 AI 架构

系统架构概览

智能简历筛选平台的 AI 子系统采用异步流水线架构，将简历从上传到最终评分拆解为多个独立的处理阶段，每个阶段可独立扩缩容、独立重试。

[ 数据接入层 ] 手动上传 / BOSS直聘API / 猎聘API / 前程无忧API / 内部推荐 | v [ 文件解析层 ] PDF Parser | Word Parser | OCR Engine(Tesseract) | v [ 数据清洗层 ] 格式标准化 → 字段提取 → 内容去噪 → 结构化转换 → 质量校验 → 脱敏处理 | v [ AI 评分层 ] Prompt 构建 → 模型路由 → LLM 调用(DeepSeek/Qwen) → 结果校验 | v [ 存储层 ] MySQL(结构化数据) | Milvus(向量索引) | MinIO(原始文件) | v [ 展示层 ] Vue 3 前端 → 评分看板 / 候选人排序 / 详情与证据

服务拆分

服务名称 职责 技术栈 部署方式

Parse Service PDF/Word/OCR 解析、文本提取、结构化转换 FastAPI + PyPDF2 + Tesseract 2 实例（CPU 密集型）

去重策略

同一候选人可能通过多个渠道投递简历，系统需要自动识别并合并：

三级去重机制

  • 精确匹配：手机号完全一致 → 确认为同一人，自动合并
  • 准精确匹配：姓名 + 邮箱一致 → 确认为同一人，自动合并
  • 模糊匹配：姓名 + 毕业院校 + 毕业年份一致 → 标记为疑似重复，HR 人工确认

合并策略：保留最新版本简历作为主档案，其他版本作为历史记录。渠道来源信息全部保留，便于分析各渠道的简历质量和转化率。

§ 09

文本提取与预处理

设计理念：让 LLM 做重活，预处理只做最少必要工作。
传统简历筛选系统需要复杂的数据清洗（正则提取、NER实体识别、字段标准化等），因为它们依赖关键词匹配。而本系统采用 LLM 方案，大模型本身就能理解排版混乱、格式不统一的原始文本，因此不需要传统意义上的数据清洗。预处理只需完成两件事：把文件变成文本、把隐私信息脱敏。

处理流程（仅 3 步）

文件 → 文本提取 → PII 脱敏 → 发送给 LLM 评分
步骤 处理内容 技术手段 说明
Step 1：文本提取 从文件中提取出纯文本内容 PDF → pdfplumber
Word → python-docx
图片/扫描件 → Tesseract OCR		 这是唯一的"转换"步骤，目标是把任何格式的简历变成文本字符串。不需要做字段拆分或结构化——LLM 会自己理解。
Step 2：PII 脱敏 在发送 LLM 前，隐藏候选人隐私信息 正则匹配（手机号、邮箱、身份证号）+ 姓名替换为"某候选人" 脱敏后的文本发给 LLM 评分，原始信息保留在本地数据库中供 HR 查看。管理员可在系统设置中配置脱敏范围。
Step 3：发送 LLM 将脱敏后的简历原文 + JD 文本一起发送给大模型 LangChain API 调用 LLM 在一次调用中完成全部工作：理解简历内容、提取关键信息、对比 JD 需求、输出结构化评分。无需人工预处理字段。

为什么不需要传统数据清洗？

传统系统需要做的 本系统是否需要 原因
正则提取姓名/手机/邮箱 仅脱敏需要，不需要结构化 LLM 直接从原文理解候选人信息
NER 实体识别（公司名、学校名） 不需要 LLM 原生理解这些实体
技能关键词提取与标准化 不需要 LLM 能理解"Spring Boot"和"SpringBoot"是同一技术
工作经历结构化拆分 不需要 LLM 能从自由文本中理解时间线和职责
去除页眉页脚/水印 不需要 LLM 会自动忽略无关内容，不影响评分
编码/格式统一 不需要 文本提取工具已输出统一的 UTF-8 字符串

文本提取失败处理

文本提取是唯一可能失败的环节。失败时按以下策略降级：

正常提取文本 → 失败? OCR 增强模式（高DPI+去噪+校正） → 仍失败? 标记"解析失败" → HR 选择：重试 / 人工录入
失败类型原因处理方式
格式无法识别扫描图片分辨率过低可重试（OCR增强模式），或人工录入关键信息
文件损坏上传不完整或文件本身损坏仅支持人工录入
文件加密PDF 设置了密码保护仅支持人工录入
内容为空文件可打开但无有效内容仅支持人工录入

人工录入时，HR 填写姓名、经验年限、核心技能等关键字段，LLM 基于这些简要信息进行评分（准确度会低于完整简历，系统会标注"信息完整度较低"）。

数据质量指标

≥98% 文本提取成功率
≤2% 需人工兜底比例
≤1% 脱敏遗漏率

Prompt 工程设计

System Prompt 结构

AI 评分的核心在于 Prompt 的设计。我们的 System Prompt 由四个部分组成：角色定义、任务描述、输出格式约束、防幻觉规则。

// ========== System Prompt 模板 ========== 【角色定义】 你是一位拥有 15 年以上经验的资深 HR 和技术面试官，专注于IT互联网行业的 人才评估。你擅长从简历文本中识别候选人的真实技术水平、项目经验深度和 职业发展潜力。你的评估风格严谨、客观、有据可依。 【任务描述】 请根据以下「岗位描述(JD)」和「候选人简历」，从 {N} 个维度对候选人进行 全面评估。每个维度给出 0-10 分的评分，并引用简历中的具体文字作为评分依据。 【评分维度】 {动态注入：根据岗位配置的维度列表、权重和评估标准} 【输出格式】 严格按照以下 JSON 格式输出，不要输出任何 JSON 之外的内容： { "dimensions": [ { "name": "维度名称", "score": 0-10的整数, "evidence": "引用简历原文作为评分依据", "comment": "评分理由的简要说明" } ], "risks": ["识别到的风险点列表"], "highlights": ["候选人的突出亮点"], "overall_comment": "对候选人的综合评价（2-3句话）" } 【约束规则】 1. 每个维度的 evidence 字段必须直接引用简历中的原文，不可编造。 2. 如果简历中缺少某维度的信息，score 给 3 分并在 comment 中说明"简历未提供相关信息"。 3. 不要假设简历未提及的能力，仅基于简历文本进行评估。 4. risks 字段需要识别：频繁跳槽、职业空白期、技能与岗位不匹配等风险。 5. 输出必须是合法的 JSON，不包含任何额外文本或 markdown 标记。

User Prompt 模板

// ========== User Prompt 模板 ========== 【岗位描述 JD】 岗位名称：{position_title} 岗位级别：{position_level} 部门：{department} --- {jd_text_compressed} 【候选人简历】 （以下信息已脱敏处理） --- {resume_text_desensitized} 请严格按照 System Prompt 中定义的格式进行评估并输出 JSON。

Few-shot 校准示例

为了确保评分一致性，在 System Prompt 中嵌入 2-3 个标注过的简历评分示例，让模型理解评分尺度：

Few-shot 策略 嵌入一个高分示例（8-9 分段）和一个低分示例（3-4 分段），让模型建立评分锚点。示例选取自评测集中 HR 共识度最高的案例。Few-shot 示例随维度配置变化自动更新。

防幻觉机制

四重防幻觉保障

  • 证据强制引用：每个维度评分必须附带简历原文引用，后端校验 evidence 是否真实出现在简历文本中
  • 低温采样：temperature 设为 0.1，最大程度降低随机性，保证相同输入产生相似输出
  • 缺失信息兜底：明确要求模型在信息缺失时给出固定低分（3 分），而非编造能力
  • 输出校验：后端对 JSON 结构、分数范围（0-10）、必填字段进行严格校验，不合规则重试

Token 优化策略

为控制 API 成本和响应速度，对输入内容进行压缩优化：

  • JD 压缩：仅保留核心技术要求、经验要求、岗位职责，移除公司介绍、福利待遇等无关内容
  • 简历压缩：工作经历保留最近 3 段，教育经历保留最高学历，移除自我评价等低信息量内容
  • 格式精简：移除多余空行、缩进、修饰性文字，保留纯信息内容
  • 目标：单份简历的总输入 tokens 控制在 2,000 以内

简历解析 Pipeline 详解

文件类型检测与路由

系统通过文件 MIME Type 和扩展名双重判断文件类型，自动选择对应的解析器：

文件类型 解析工具 优势 限制
PDF（文本型） pdfplumber 保留表格结构、精确提取文本和坐标 无法处理图片型 PDF
PDF（扫描型） Tesseract OCR 支持中英文混合识别，可配置 DPI 依赖扫描质量，低分辨率可能失败
Word（docx） python-docx 完美保留文档结构和格式信息 不支持 doc 格式（需先转换）
图片（jpg/png） Tesseract OCR 直接对图片进行文字识别 手写体、艺术字体识别率较低
加密 PDF —— —— 自动检测加密，标记为"文件加密"

OCR 增强模式

当标准解析失败时，系统自动切换到 OCR 增强模式进行重试：

OCR 增强处理流程

  • 高 DPI 重渲染：将 PDF 页面以 300 DPI 重新渲染为图片（标准模式为 150 DPI）
  • 图像预处理：对比度增强、降噪（高斯模糊 + 二值化）、倾斜矫正（Hough 变换）
  • 分区识别：将页面分割为多个区域，逐区域识别后合并，提高复杂版面的准确率
  • 置信度过滤：OCR 结果的字符置信度低于 60% 的区域标记为"识别不确定"

解析失败分类与处理

失败类别 错误码 触发条件 处理策略
格式错误 FORMAT_ERROR 低分辨率扫描件、严重损坏的排版 OCR 增强重试 → 人工录入
文件损坏 CORRUPT_FILE 文件无法打开、数据流异常 直接转人工录入
文件加密 ENCRYPTED PDF 设有密码保护 通知 HR 获取无密码版本或人工录入
内容为空 EMPTY_CONTENT 文件能打开但提取不到任何文字 OCR 增强重试 → 人工录入

输出格式

解析成功后，输出标准化的 Resume JSON 对象，包含以下顶层字段：

{ "parse_meta": { "parser": "pdfplumber", "parse_time_ms": 2340, "confidence": 0.95, "ocr_used": false, "pages": 2 }, "personal": { ... }, "education": [ ... ], "work_history": [ ... ], "skills": [ ... ], "projects": [ ... ], "raw_text": "完整提取文本（用于LLM评分）", "quality_flags": ["MISSING_PHONE", "SHORT_WORK_HISTORY"] }
§ 13

评分算法详解

评分流程

从结构化简历数据到最终评分结果，需要经过五个步骤的计算：

Step 1: LLM 维度评分 → Step 2: 加权计算 → Step 3: 红线惩罚 → Step 4: 归一化 → Step 5: 等级判定

Step 1：LLM 维度评分

将脱敏后的简历文本和 JD 发送给 LLM，获取每个维度的独立评分（0-10 整数）和评分证据。

Step 2：加权计算

// 加权平均计算 weighted_sum = 0 weight_total = 0 for dim in dimensions: weighted_sum += dim.score * dim.weight weight_total += dim.weight base_score = weighted_sum / weight_total // 结果为 0-10 区间 // 示例： // 技术深度: 8分 x 30% = 2.4 // 项目经验: 7分 x 25% = 1.75 // 行业理解: 6分 x 15% = 0.9 // 团队协作: 7分 x 15% = 1.05 // 学习潜力: 8分 x 15% = 1.2 // base_score = (2.4+1.75+0.9+1.05+1.2) / 1.0 = 7.3

Step 3：红线惩罚

如果候选人触发了红线规则，按照以下规则进行惩罚扣分：

红线类型 惩罚方式 扣分值
经验年限不足 绝对扣分 -20 分（百分制）
学历不达标 绝对扣分 -20 分（百分制）
工作稳定性差 绝对扣分 -15 分（百分制）
黑名单命中 直接归零 score = 0

Step 4：归一化至 0-100

// 归一化公式 final_score = max(0, round(base_score * 10) - penalty) // 完整公式表达： // score = max(0, round( SUM(dim_score_i * weight_i) / SUM(weight_i) * 10 ) - penalty) // 示例（无红线惩罚）： // final_score = max(0, round(7.3 * 10) - 0) = 73 // 示例（触发经验不足红线）： // final_score = max(0, round(7.3 * 10) - 20) = 53

Step 5：等级判定

最终得分 等级标签 颜色标识 HR 操作建议
85-100 强烈推荐 红色 优先安排面试，48 小时内联系
75-84 推荐 绿色 纳入面试候选池
60-74 建议复核 黄色 HR 人工复核评分理由后决定
0-59 不推荐 灰色 默认不安排面试，可"强捞"

HR 评分解读指南

如何使用 AI 评分 AI 评分是辅助工具而非最终决策。建议 HR 重点关注：(1) 各维度评分是否均衡，有无明显短板；(2) evidence 引用的简历原文是否准确反映候选人水平；(3) risks 列表中是否有不可接受的风险点。对于"建议复核"区间的候选人，应仔细阅读 AI 给出的评分理由后再做判断。

灰度对比测试

  • 流量分割：按候选人 ID 哈希分流，确保同一候选人在测试期间始终走同一版本
  • 双版本并跑：实验组和对照组同时记录评分结果，HR 仅看到其中一组
  • 统计显著性：至少运行 50 个样本后，用卡方检验判断差异是否显著（p < 0.05）
  • 自动决策：实验组所有指标优于对照组且统计显著时，自动推荐全量上线

持续评估

除了变更前的回归测试，系统还执行月度持续评估：

  • 每月抽检：随机抽取 50 份已评分简历，由资深 HR 重新独立打分
  • 漂移检测：对比 AI 评分与人工重评结果，监测评分是否出现系统性偏移
  • 评估报告：每月生成质量评估报告，包含准确率变化趋势、错误案例分析、优化建议
§ 15

模型自迭代与持续优化

每日自动优化

系统通过每日自动化流程持续优化评分质量，形成"数据驱动 + 安全约束"的自迭代机制。

阶段 执行时间 操作内容 安全约束
数据采集 2:00 AM 收集前一天所有 HR 修正操作：强捞低分候选人、否决高分候选人、手动调分记录 仅采集已确认的操作
偏差分析 2:10 AM 统计各维度的系统性偏差方向和幅度，识别需要调整的维度 至少 5 条修正记录才触发分析
权重微调 2:20 AM 对偏差维度进行权重微调，单次调整幅度不超过 ±2% 安全护栏 单次 ≤2%，累计 ≤10%
回归验证 2:30 AM 在 200 份评测集上运行回归测试，对比调整前后各项指标 任一核心指标下降则不部署
自动部署 2:45 AM 验证通过则自动部署新规则版本，否则保持原版本 自动生成变更日志
安全护栏设计 自动优化的核心风险是过度拟合或误调。系统通过以下护栏确保安全：(1) 单次权重调整不超过 ±2%；(2) 累计自动调整不超过初始值的 ±10%，超出需人工审批；(3) 连续 3 天回归不通过时，自动暂停自优化并通知产品团队。

月度人工审查

月度质量回顾

  • 参与人员：产品经理、HR 负责人、技术负责人
  • 审查内容：月度评分质量报告、HR 反馈汇总、维度权重变化趋势
  • 决策权限：可审批超过 ±10% 的权重调整、新增/删除评分维度、修改 Prompt 模板
  • 输出物：更新后的评分规则版本、下月优化方向

§ 16

向量库与语义检索

Milvus 向量数据库

系统采用 Milvus 2.3 作为向量数据库，支撑三个核心场景的语义检索需求。

三大应用场景

场景 输入 检索方式 阈值 用途
简历去重 新简历向量 最近邻搜索 + 余弦相似度 cosine ≥ 0.92 判定是否同一人，防止重复处理
相似推荐 目标候选人向量 Top-K 近邻（K=20） cosine ≥ 0.75 "帮我找更多类似这位候选人的简历"
人才库搜索 自然语言查询向量 向量搜索 + 属性过滤 cosine ≥ 0.60 HR 用自然语言描述需求，语义搜索候选人库

向量化方案

Embedding 技术选型

  • Embedding 模型：text-embedding-v2（通义千问系列），768 维向量
  • 输入内容：脱敏后的简历全文（截断至 512 tokens），重点保留技能和经验描述
  • 索引类型：IVF_FLAT（倒排文件 + 扁平量化），在检索速度和准确率之间取得平衡
  • nlist 参数：128（适合万级数据规模，500-10000 条简历）
  • nprobe 参数：16（搜索时探测 16 个聚类中心，召回率约 95%）

向量更新策略

  • 新增：简历解析完成后，立即生成向量并写入 Milvus
  • 更新：简历内容发生变更时（如 HR 手动编辑），重新生成向量并替换
  • 删除：候选人从系统中移除时，同步删除对应向量
  • 批量重建：Embedding 模型升级时，可触发全量向量重建任务（预计耗时 30 分钟/1000 条）

性能指标

<50ms 单次检索延迟
≥95% 召回率
768 向量维度
10K 设计容量上限
§ 17

性能与并发设计

异步处理架构

简历从上传到评分完成是一个耗时过程（可能需要 30 秒到 2 分钟），系统采用全异步架构，确保用户体验不受影响。

上传文件 → 立即返回 task_id → 后台解析 → 后台评分 → WebSocket 通知完成

用户上传简历后立即收到上传成功响应（含任务 ID），后续的解析和评分在后台异步执行。当处理完成时，通过 WebSocket 实时推送通知到前端。

任务队列设计

队列名称 优先级 处理场景 并发数
parse:high 高 HR 手动上传的简历 5
parse:normal 中 自动同步的简历（BOSS/猎聘/前程无忧） 3
score:high 高 手动触发的评分请求 5
score:normal 中 自动批量评分 3
score:rescore 低 规则变更后的批量重评 2

批量处理

批量上传处理规则

  • 单批上限：50 份简历/次，超出部分自动排队到下一批
  • 并行处理：同批简历最多 5 份并行解析，避免资源竞争
  • 进度反馈：前端显示批量处理进度条（已完成/总数），支持取消未处理的剩余任务
  • 失败隔离：单份简历失败不影响同批其他简历的处理

扩缩容策略

当前规模（500 份/年）下，2 实例足以支撑。但系统设计了水平扩缩容能力，以应对招聘旺季或未来业务增长：

  • 扩容触发：当任务队列深度超过 100 且持续 5 分钟时，自动扩容 AI 服务实例
  • 缩容触发：当队列深度持续为 0 超过 30 分钟时，缩减至最小实例数
  • 最小实例：2 个（保证高可用）
  • 最大实例：6 个（成本上限约束）
